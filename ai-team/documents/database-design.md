# 🗄️ Taroudant Heritage Shield — Database Design

## Database: MySQL | Normalization: 3NF

### What is 3NF?
Every non-key attribute must depend on:
- The key (1NF — atomic values, no arrays)
- The whole key (2NF — no partial dependencies)
- Nothing but the key (3NF — no transitive dependencies)

---

## Tables Overview

### 1. roles
Stores the 4 system roles for RBAC.
Fields: role_id, role_name, description
3NF note: Extracted from users to avoid transitive dependency.

### 2. users
All system actors — admin, inspector, authority, viewer.
Fields: id_user, full_name, password_hash, role_id (FK),
        phone, email, created_at
3NF note: role_name lives in roles table, not here.

### 3. monument_categories
Classification of monument types (mosquée, rempart, fontaine...)
Fields: category_id, category_name, description
3NF note: Extracted from monuments to avoid transitive dependency.

### 4. monuments
The heart of the system. Every other table references this.
Fields: monument_id, name, location, city, latitude, longitude,
        construction_year, category_id (FK), description, status
3NF note: category details live in monument_categories, not here.

### 5. monument_assets
Photos and media — each asset is its own row (1NF compliance).
Fields: photo_id, monument_id (FK), photo_url, caption,
        uploaded_at, uploaded_by (FK → users)
3NF note: Separated from monuments — a monument can have many photos.

### 6. inspections
Records when and by whom a monument was inspected.
Fields: inspection_id, monument_id (FK), inspector_id (FK → users),
        inspection_date, notes, overall_condition
3NF note: Inspector details live in users, not duplicated here.

### 7. cracks
Each crack detected is its own row — feeds the scoring engine.
Fields: crack_id, inspection_id (FK), location_on_monument,
        severity (minor/moderate/major/critical),
        length_cm, photo_url, detected_at
3NF note: 1NF compliance — no comma-separated crack lists.

### 8. vulnerability_scores
Computed output of the scoring stored procedure.
Fields: score_id, monument_id (FK), inspection_id (FK),
        age_score, crack_score, total_score, risk_level, computed_at
3NF note: Stores snapshot values — not recalculated on every read.

### 9. notifications
Auto-generated alerts from triggers + manual assignment notifications.
Fields: notification_id, monument_id (FK),
        triggered_by_inspection (FK), recipient_id (FK → users),
        message, severity, is_read, sent_at

### 10. reports
Encrypted expert reports generated by inspectors.
Fields: report_id, monument_id (FK), inspection_id (FK),
        generated_by (FK → users), title, file_path,
        risk_level, total_score, status, created_at
Security: Report content encrypted with AES-256.
          Encryption key stored server-side, never in DB.

### 11. audit_logs
Tracks every sensitive action for accountability and security.
Fields: log_id, user_id (FK), action, target_table,
        target_id, ip_address, details, performed_at
Examples of logged actions:
  REPORT_GENERATED, REPORT_ACCESSED, CRACK_LOGGED,
  USER_CREATED, SCORE_UPDATED, LOGIN_SUCCESS, LOGIN_FAILED

## Business Logic (Stored Procedures & Triggers)

### Stored Procedure 1: CalculateVulnerabilityScore
- Input: inspection_id
- Calculates age_score + crack_score → assigns risk_level
- Output: INSERT into vulnerability_scores

### Stored Procedure 2: GenerateMonumentReport
- Input: monument_id, inspection_id, generated_by
- Compiles all inspection data → encrypts → saves report
- Output: INSERT into reports

### Trigger 1: after_crack_insert
- Fires: AFTER INSERT on cracks
- Action: Calls CalculateVulnerabilityScore automatically

### Trigger 2: after_score_insert
- Fires: AFTER INSERT on vulnerability_scores
- Action: If risk_level is high or critical →
          INSERT notifications for all authority users

### Trigger 3: after_report_insert
- Fires: AFTER INSERT on reports
- Action: INSERT into audit_logs (action: REPORT_GENERATED)

## Security Summary

| Threat | Protection |
|---|---|
| SQL Injection | Prepared statements in all backend queries |
| Unauthorized access | JWT + RBAC middleware on every route |
| Sensitive report leak | AES-256 encryption, key never in DB |
| Untracked access | audit_logs records every sensitive action |
| Weak passwords | bcrypt hashing — plain passwords never stored |
